## 1. ✅ Безопасные оптимизации (можно применять сразу)

### 1.1 CPU Architecture: GENERIC → x86-64-v2 (или MCORE2)

**Что:** Заменить `CONFIG_GENERIC_CPU=y` на `CONFIG_MCORE2=y`  
**Где:** `.config`  
**Почему:** Sandy Bridge поддерживает SSE4.2, POPCNT, но **не AVX2/AVX-512**. `GENERIC_CPU` даёт минимум (x86-64-v1). `MCORE2` (x86-64-v2) включает оптимизации для Core 2 и новее — SSE4.2, POPCNT.  
**Риски:** Отсутствуют — всё это железо поддерживает.  
**Ожидание:** +3-5% CPU performance  
**Откат:** `grep -v CONFIG_MCORE2 .config && echo CONFIG_GENERIC_CPU=y >> .config`

### 1.2 NR_CPUS: 8192 → 8

**Что:** Уменьшить `CONFIG_NR_CPUS=8192` до `CONFIG_NR_CPUS=8`  
**Где:** `.config`  
**Почему:** У вас 4 ядра / 8 потоков. NR_CPUS=8192 выделяет структуры планировщика на 8192 CPU, что тратит ~2-4MB лишней памяти на scheduler bitmaps.  
**Риски:** Если никогда не планируете ставить 8-сокетный сервер — отсутствуют.  
**Ожидание:** Экономия памяти, микроускорение планировщика  
**Откат:** `scripts/config --set-val NR_CPUS 8192`

### 1.3 MAXSMP → отключить

**Что:** Отключить `CONFIG_MAXSMP=y`  
**Где:** `.config`  
**Почему:** MAXSMP включает оптимизации для систем >8 CPU. На 8 потоках даёт обратный эффект.  
**Риски:** Нет.  
**Ожидание:** -0.5% latency в планировщике

### 1.4 PCIe ASPM: DEFAULT → PERFORMANCE

**Что:** `CONFIG_PCIEASPM_DEFAULT=y` → `CONFIG_PCIEASPM_PERFORMANCE=y`  
**Где:** `.config`  
**Почему:** ASPM в режиме PERFORMANCE отключает энергосбережение PCIe, снижая latency для GPU и NVMe.  
**Риски:** +2-3 Вт энергопотребления (неактуально для десктопа).  
**Ожидание:** -1ms GPU latency, +3% NVMe throughput  
**Откат:** `scripts/config -e PCIEASPM_DEFAULT`

### 1.5 DEBUG_INFO: DWARF → ZSTD сжатие

**Что:** `CONFIG_DEBUG_INFO_COMPRESSED_NONE` → `CONFIG_DEBUG_INFO_COMPRESSED_ZSTD`  
**Где:** `.config`  
**Почему:** DWARF5 отладочная информация без сжатия занимает ~300MB. ZSTD сжимает до ~40MB без потери информации. BTF (BPF Type Format) остаётся.  
**Риски:** Чуть медленнее распаковка при отладке. На проде не используется.  
**Ожидание:** Экономия ~250MB на vmlinux

### 1.6 EFI_MIXED → отключить

**Что:** `# CONFIG_EFI_MIXED is not set`  
**Где:** `.config`  
**Почему:** Вы грузитесь с 64-битного UEFI. EFI_MIXED — поддержка 32-битного EFI на 64-битном ядре.  
**Риски:** Нет, если не используете 32-битный UEFI.  
**Ожидание:** -0.1% размер ядра

### 1.7 Подпись модулей: SHA512 → SHA256 (быстрее)

**Что:** `CONFIG_MODULE_SIG_SHA512` → `CONFIG_MODULE_SIG_SHA256`  
**Где:** `.config`  
**Почему:** SHA-256 в ~2x быстрее SHA-512 на процессорах без аппаратного ускорения SHA (Sandy Bridge не имеет SHA-NI). Безопасность SHA-256 достаточна для подписи модулей.  
**Риски:** Нет, SHA-256 криптостойкий.  
**Ожидание:** Быстрая загрузка модулей

### 1.8 Отключить ненужные Security LSM

**Что:** Убрать Smack, Tomoyo, LoadPin, SafeSetID, IPE  
**Где:** `.config`  
**Почему:** На домашнем ПК с AppArmor (или SELinux) остальные LSM не используются, но тратят память.  
**Риски:** Минимальные — оставить AppArmor и/или SELinux.  
**Ожидание:** Экономия ~1MB, быстрее загрузка

## 2. ⚠️ Рекомендуемые оптимизации (нужно тестирование)

### 2.1 NUMA → отключить

**Что:** `# CONFIG_NUMA is not set`  
**Где:** `.config`  
**Почему:** Sandy Bridge Xeon E3 — однопроцессорный, без NUMA. NUMA балансинг добавляет ~1-3% накладных расходов на каждом page fault.  
**Риски:** Если у вас на самом деле multi-socket (маловероятно для E3) — упадёт производительность.  
**Ожидание:** +1-3% на память-интенсивных задачах  
**Откат:** `scripts/config -e NUMA`

### 2.2 PREEMPT → PREEMPT_LAZY

**Что:** `CONFIG_PREEMPT` → `CONFIG_PREEMPT_LAZY`  
**Где:** `.config`  
**Почему:** PREEMPT_LAZY — новый компромисс между PREEMPT (полная вытесняемость) и PREEMPT_VOLUNTARY. Даёт лучшую пропускную способность для CPU-bound задач, сохраняя низкую задержку для интерактива.  
**Риски:** Возможно микро-заикания в играх (нужно тестировать).  
**Ожидание:** +2-5% throughput, те же latency  
**Тест:** `stress-ng --cpu 8 --timeout 30s` + замер FPS в игре

### 2.3 THP: ALWAYS → MADVISE

**Что:** `TRANSPARENT_HUGEPAGE_ALWAYS` → `TRANSPARENT_HUGEPAGE_MADVISE`  
**Где:** `.config` + PKGBUILD  
**Почему:** CachyOS включает ALWAYS для huge pages. Но на Rust-программах, Electron (браузер), и некоторых БД это может увеличивать latency из-за khugepaged дефрагментации. MADVISE отдаёт выбор приложению.  
**Риски:** Может незначительно снизить производительность в некоторых синтетиках.  
**Ожидание:** Более стабильный latency  
**Тест:** Замерить `khugepaged` CPU usage через `htop` в простое

### 2.4 NUMA_BALANCING → отключить

**Что:** `# CONFIG_NUMA_BALANCING is not set`  
**Где:** `.config`  
**Почему:** Если NUMA отключён (см. 2.1), нума-балансинг бессмыслен.  
**Риски:** Зависит от NUMA.  
**Ожидание:** +1-2% на CPU-bound задачах

### 2.5 SCHED_CLASS_EXT → отключить (если не используете sched_ext)

**Что:** `# CONFIG_SCHED_CLASS_EXT is not set`  
**Где:** `.config`  
**Почему:** sched_ext — для кастомных BPF-планировщиков (scx_*). Если вы не используете `scx_rusty` или аналог — это лишняя вложенность в планировщике.  
**Риски:** Если используете кастомный планировщик — сломается.  
**Ожидание:** Микроускорение планировщика

## 3. 🧪 Экспериментальные оптимизации

### 3.1 GCC с PGO (Profile Guided Optimization)

**Что:** Собрать ядро с GCC + PGO через `make -j$(nproc) && make -j$(nproc) pgo_profile`  
**Где:** PKGBUILD  
**Почему:** PGO позволяет компилятору оптимизировать горячие пути по реальному профилю выполнения.  
**Риски:** Нужен тренировочный прогон. Если профиль не репрезентативен — оптимизация может навредить.  
**Ожидание:** +5-10% CPU-bound задач  
**Сложность:** Высокая (нужен 2-этапный билд)

### 3.2 BOLT (Binary Optimization and Layout Tool)

**Что:** Применить BOLT после сборки для оптимизации расположения кода  
**Где:** После билда  
**Почему:** BOLT переупорядочивает basic blocks в памяти для лучшего использования i-cache.  
**Риски:** Нестабильность на некоторых версиях ядра.  
**Ожидание:** +2-5%

### 3.3 AUTOFDO (Auto-Feedback-Directed Optimization)

**Что:** Использовать авто-FDO для автоматического сбора профиля  
**Где:** PKGBUILD  
**Почему:** Похож на PGO, но автоматический сбор профиля после загрузки.  
**Риски:** Экспериментальная поддержка в ядре.  
**Ожидание:** +2-8%

### 3.4 Тонкая настройка ZSWAP/ZRAM

**Что:** `CONFIG_ZSWAP_SHRINKER_DEFAULT_ON=y` → отключить  
**Где:** `.config`  
**Почему:** На 16GB RAM shrinker ZSWAP может вызывать лишние CPU cycles.  
**Риски:** При нехватке памяти — OOM.  
**Ожидание:** -1-2% CPU usage, больше свободной памяти

### 3.5 Полная кастомизация .config

**Что:** Вручную отключить ~1000 ненужных драйверов  
**Где:** `.config`  
**Почему:** Каждый встроенный драйвер, даже без устройств, тратит память.  
**Риски:** Очень высокие — легко отключить нужное.  
**Ожидание:** Экономия ~50MB RAM, +1-2% скорость загрузки

## 4. ❌ Не рекомендуемые оптимизации

### 4.1 Полное отключение DEBUG_INFO

**Почему:** BTF нужен для modern eBPF (bpftrace, tracing, сетевые фильтры). Без BTF многие инструменты не работают.

### 4.2 Отключение MODULE_SIG

**Почему:** Без подписи можно загрузить вредоносный модуль ядра. Риск не стоит микро-выигрыша.

### 4.3 MITIGATIONS = OFF

**Почему:** Spectre/Meltdown/L1TF уязвимости на Sandy Bridge реальны. `mitigations=off` открывает систему.

### 4.4 CONFIG_GENERIC_CPU2 / x86-64-v3

**Почему:** x86-64-v3 требует AVX2. Sandy Bridge его не имеет. Ядро упадёт с SIGILL.

### 4.5 PSI (Pressure Stall Information) отключение

**Почему:** PSI даёт важную информацию о загрузке системы. Экономия микроскопическая.

## 5. Оптимизации для PKGBUILD (кроме .config)

### 5.1 ✅ Добавить `march=native` или `-march=x86-64-v2`

```bash
export CFLAGS="-march=x86-64-v2 -O3 -pipe -fno-plt -fgraphite-identity -floop-nest-optimize"
export CXXFLAGS="${CFLAGS}"
```

**Где:** PKGBUILD, в функцию `build()`  
**Почему:** `-O3` + `-march=x86-64-v2` включит оптимизации для Sandy Bridge.  
**⚠️ ВАЖНО:** CachyOS использует Clang, не GCC. Для Clang флаг `-march=native` или `-march=x86-64-v2`.

### 5.2 ✅ Добавить `-flto=thin` (уже есть в .config)

**Статус:** ✅ УЖЕ ВКЛЮЧЁН

### 5.3 ⚠️ Использовать `make -j$(nproc) > /dev/null` для чистого вывода

**Где:** PKGBUILD  
**Почему:** Логи сборки меньше, makepkg быстрее завершается.
